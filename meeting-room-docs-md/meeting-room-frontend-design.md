# 会议室预定系统前端设计文档

## 1. 设计目标
前端设计文档聚焦于“如何实现”会议室预定系统的用户端与管理端界面，包括页面拆分、组件设计、状态流转、接口对接、权限控制、可维护性和性能策略。

## 2. 总体设计原则
- 以“快速找房 + 快速预定”为核心
- 以组件化方式降低维护成本
- 以状态驱动界面变化
- 以统一数据模型减少前后端沟通成本
- 兼顾桌面端主流程与移动端扩展

## 3. 前端架构设计
### 3.1 分层结构
```text
表现层（Pages / Components）
    ↓
业务层（Hooks / Store / Services）
    ↓
接口层（API）
    ↓
基础层（Utils / Types / Constants）
```

### 3.2 模块划分
- auth：登录与权限
- room：会议室列表、详情、筛选
- booking：预定创建、修改、取消
- calendar：日历视图、时间轴
- admin：后台管理
- stats：统计分析
- common：通用组件、枚举、工具方法

## 4. 页面设计
## 4.1 登录页
### 功能
- 用户认证
- 获取用户角色与权限
- 初始化全局配置

### 交互要点
- 登录成功跳转首页
- 登录失败提示明确
- 支持回车提交

## 4.2 首页 / 工作台
### 模块
- 快速筛选区
- 当前推荐会议室
- 今日我的会议
- 通知公告

### 设计要点
- 首屏应提供最短路径进入“查找空闲会议室”
- 推荐房间可基于最近使用、容量匹配、设备匹配进行展示

## 4.3 会议室列表页
### 布局
- 顶部筛选区
- 左侧附加筛选（可选）
- 中间结果列表
- 右侧地图/楼层图（可选）

### 数据展示
每个会议室卡片包含：
- 名称
- 楼宇/楼层
- 容量
- 设备标签
- 当前状态
- 最近可用时间
- “查看详情”与“立即预定”按钮

### 筛选条件状态
```ts
type RoomFilter = {
  date: string
  startTime: string
  endTime: string
  capacity?: number
  building?: string
  floor?: string
  facilities?: string[]
  status?: 'AVAILABLE' | 'BUSY' | 'MAINTENANCE'
}
```

## 4.4 会议室详情页
### 页面区块
- 基础信息区
- 图片 / 平面图区
- 时间轴占用区
- 预定入口区
- 历史信息 / 注意事项区（可选）

### 设计重点
- 时间轴必须足够直观，用户能快速判断可预定区间
- 对被占用时段展示会议标题时需考虑隐私脱敏策略

## 4.5 我的预定页
### 页面功能
- 列表查看
- 状态筛选
- 详情查看
- 修改 / 取消 / 再次预定

### 展示字段
- 会议主题
- 会议室名称
- 日期与时间
- 预定状态
- 签到状态
- 操作按钮

## 4.6 管理后台页面
### 功能模块
- 会议室管理表格
- 规则配置表单
- 预定记录查询
- 数据图表分析

## 5. 核心组件设计
## 5.1 RoomCard
### 作用
在列表中展示会议室概要信息。

### Props
```ts
interface RoomCardProps {
  id: number
  name: string
  building: string
  floor: string
  capacity: number
  facilities: string[]
  status: string
  availableSlots?: string[]
}
```

### 事件
- onViewDetail
- onBookNow

## 5.2 BookingForm
### 作用
新建或编辑预定。

### 字段
- subject
- date
- startTime
- endTime
- attendeeCount
- roomId
- remark

### 校验规则
- 主题不能为空
- 开始时间必须小于结束时间
- 结束时间不得超出营业时间
- 人数不得超过会议室容量

## 5.3 TimeLine / ScheduleGrid
### 作用
展示会议室一天的占用情况。

### 设计建议
- 以 30 分钟粒度渲染
- 支持 hover 显示详情
- 支持点击空白区域快速进入预定

## 5.4 FilterBar
### 功能
统一承载时间、人数、设备、楼层筛选。

### 设计原则
- 高优先级条件放前面：日期、时间、人数
- 高级筛选可折叠收起

## 5.5 StatusTag
统一封装状态标签，避免不同页面状态颜色不一致。

## 6. 状态管理设计
## 6.1 全局状态
适合存储：
- 当前用户信息
- 权限列表
- 系统配置
- 楼宇/楼层基础字典
- 全局 loading 状态

## 6.2 模块状态
### room 模块
- 当前筛选条件
- 查询结果列表
- 分页信息
- 当前选中会议室

### booking 模块
- 当前编辑中的表单数据
- 冲突校验结果
- 当前操作状态（提交中/成功/失败）

### admin 模块
- 当前表格条件
- 当前编辑中的会议室对象

## 6.3 状态管理建议
- 全局共享的数据进入 Store
- 页面局部表单状态优先使用组件内状态
- 避免所有数据都放入全局 Store，降低复杂度

## 7. 路由设计
```text
/login
/home
/rooms
/rooms/:id
/bookings/new
/bookings/my
/calendar
/admin/rooms
/admin/bookings
/admin/rules
/admin/stats
```

### 路由守卫
- 未登录用户跳转登录页
- 无权限用户跳转 403 页面
- 登录状态失效则自动清理本地缓存并跳转

## 8. 接口对接设计
## 8.1 接口分层建议
```text
api/
  auth.ts
  room.ts
  booking.ts
  admin.ts
  stats.ts
```

## 8.2 统一响应格式
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 8.3 主要接口
### 查询空闲会议室
`GET /api/rooms/available`

请求参数：
- date
- startTime
- endTime
- capacity
- building
- facilities

### 获取会议室详情
`GET /api/rooms/{id}`

### 获取会议室排期
`GET /api/rooms/{id}/schedule`

### 创建预定
`POST /api/bookings`

### 修改预定
`PUT /api/bookings/{id}`

### 取消预定
`POST /api/bookings/{id}/cancel`

### 获取我的预定
`GET /api/bookings/my`

## 9. 权限设计
### 9.1 前端权限控制维度
- 路由权限
- 菜单权限
- 按钮权限
- 字段权限（如管理员可见维护原因）

### 9.2 实现建议
后端返回权限码集合，前端通过指令或 hooks 统一处理，如：
```ts
const permissions = ['room:view', 'booking:create', 'admin:room:edit']
```

## 10. 表单与交互细节设计
## 10.1 防重复提交
提交按钮在请求期间置灰并展示 loading。

## 10.2 冲突提示
当房间在提交前已被占用时，前端应提示：
- 当前房间不可用
- 推荐相同条件下的其他会议室
- 引导用户修改时间

## 10.3 时间选择联动
- 开始时间变化后自动校验结束时间
- 如果结束时间早于开始时间，则自动清空结束时间并提示

## 10.4 用户反馈
- 成功：Toast + 跳转
- 失败：错误原因明确展示
- 警告：对超时未签到等情况展示醒目提醒

## 11. 性能设计
### 11.1 页面性能
- 筛选结果分页
- 大列表虚拟滚动（如房间数量多）
- 路由级懒加载
- 图表按需加载

### 11.2 接口性能配合
- 列表页只请求必要字段
- 详情页按需补充排期与图片信息
- 对静态字典数据做本地缓存

### 11.3 用户体验优化
- 骨架屏
- 空状态页
- 接口防抖
- 筛选条件缓存恢复

## 12. 可维护性设计
### 12.1 组件复用
将会议室卡片、状态标签、时间选择区等抽象为通用组件。

### 12.2 类型统一
统一维护 `Room`、`Booking`、`User`、`Permission` 等类型定义。

### 12.3 常量统一
统一维护：
- 状态码
- 设备标签
- 楼宇字典
- 错误码与提示文案映射

## 13. 日志与监控
前端建议接入：
- 页面异常监控
- 接口失败监控
- 核心操作埋点（查询、创建预定、取消预定）
- 页面加载性能监控

## 14. 可访问性与适配
- 表单控件支持键盘操作
- 颜色标识需配合文字状态
- 重点页面支持 1366 宽度以上桌面端
- 用户端可考虑适配移动端 H5 或小程序壳

## 18. 设计结论
前端设计应围绕“查得到、看得懂、订得快、改得方便、管得清楚”五个目标展开。建议先完成用户主流程与管理后台基础能力，再逐步引入日历视图、签到、统计和智能推荐能力，从而形成一套可迭代的会议室管理前端体系。
